Introduction
What This Document Is
Purpose

This is the Dev Playbook — a standing document that defines how every software project built by Shea Gunther is structured, coded, documented, and delivered. It is not tied to any single project. It is the baseline that every project inherits.

The playbook covers four areas: process (how work is organized and tracked), code standards (how code is written so that humans can read and maintain it), quality baselines (mobile behavior, accessibility, security, performance), and records (how the history and architecture of a project are documented).

The playbook is a living document. It is updated as experience accumulates — when a new standard is adopted, when a previous rule proves wrong, or when a new category of concern is identified. The version number in the header reflects its current revision. Older versions are retained.

Instructions for the AI
To the AI reading this document

This document is being shared with you at the start of a development project. Read it fully before doing anything else. It establishes the standards you are expected to follow for the entire project, without being reminded of each rule on every turn.

Specifically:

  • Apply the versioning protocol from the first deliverable. Name every file correctly. Maintain the dev badge in the app.
  • Write all code to the Clean Code standards in Chapter 5. Do not use abbreviations. Do not write functions that do more than one thing. Do not leave magic numbers unexplained. Comments explain why, not what.
  • Apply the mobile, accessibility, security, and performance baselines in Chapters 6–9 to every app deliverable, even if not explicitly asked.
  • When a project reaches approximately 20 versions, prompt to write a dev history entry.
  • When meaningful architectural decisions are made, note them — they belong in the technical reference and dev history.
  • All accompanying documents are HTML unless explicitly requested otherwise, and use the project's established doc template (OG + Sunrise skins, sidebar TOC).
  • Do not ask for permission to follow these standards. They are always in effect.
  • If something in the playbook conflicts with an explicit instruction given during the project, follow the explicit instruction and flag the conflict.

When you have read and understood this document, acknowledge it briefly and ask what we are building.

How This Document Evolves

The playbook is versioned the same way any other deliverable is versioned — integer version numbers, incrementing by one, reflected in the file name: dev-playbook(N).html. When the playbook is updated, the old version is kept. The new version is shared with the AI on the next project.

Reasons to update the playbook include: a new standard being adopted, an existing rule being refined or reversed based on experience, a new chapter being added, or a change in the document structure itself. The update does not need to be large to justify a new version — a single meaningful clarification is enough.

Note The playbook version and the project version are independent counters. A project can be at v34 while the playbook is still at v2. The playbook version tracks the evolution of the standards themselves, not any specific project.
Chapter 1
Project Kickoff Protocol

No code gets written until the project is understood. The kickoff is the process of getting that understanding on record. It protects against building the wrong thing, and it gives the AI enough context to make good decisions throughout the project without constant re-explanation.

Required Questions

At the start of every project, the following questions must be answered before any design or code work begins. The AI is responsible for asking any that have not already been answered.

QuestionWhy It Matters
What does this app do? A one-sentence answer. If it cannot be said in one sentence, the scope is not clear yet.
Who uses it? Determines vocabulary, complexity level, and assumptions about technical comfort.
What device does it run on? Phone, desktop, tablet, or all three. Determines layout approach and touch requirements.
What does "done" look like? The minimum feature set that constitutes a usable v1. Everything else is backlog.
Where does data go? Local storage, export only, a server, nowhere. Determines storage architecture early.
Who else touches this besides the primary user? Multi-user scenarios, admin roles, staff vs. customer access — these shape auth and permissions.
Are there existing forms, processes, or documents this replaces? Understanding what currently exists prevents reinventing a workflow that already works.
For the AI Do not fire all of these as a list at once. Read the conversation for what has already been answered. Ask only what is genuinely missing. Group related questions. If the person has shared a form, a screenshot, a website, or an existing document, read it before asking questions that it might already answer.
Before Writing Code

Once the kickoff questions are answered, confirm understanding before building. This does not need to be a formal document — a brief plain-language summary of what is being built, for whom, and what the v1 feature set includes is enough. The person confirms or corrects it, and then code begins.

Common mistake Starting to code based on a partial understanding and then discovering a fundamental mismatch at v8. The kickoff exists to catch that mismatch in a two-sentence conversation, not after hours of work.
Chapter 2
Versioning Protocol

Every project uses the same version numbering and file naming system from the first deliverable onward. Anyone looking at a file name should know exactly where it sits in the project's history. Any two files with matching version numbers correspond to the same state of the work.

Version Numbers

Versions are integers, starting at 1, incrementing by 1 with every new deliverable. No major/minor splits, no semantic versioning, no decimals. The version number is a counter — it records how many times a deliverable has been produced, nothing more.

Rule A new version is produced every time a change is made and a new file is delivered. No change is too small to increment. If a file was exported, it gets a new number.
File Naming

All project files follow the same pattern: project-name(version).extension

Version number in parentheses, immediately after the base name, before the extension. No spaces. Base name uses lowercase with hyphens.

ExampleMeaning
deer-hill-intake(1).htmlFirst deliverable of the intake app
deer-hill-intake(14).htmlFourteenth version of the same app
user-manual(14).htmlUser manual built at app version 14
dev-history(20).htmlDevelopment history written at app version 20
dev-playbook(2).htmlSecond revision of this playbook
Important Accompanying documents are named with the version of the app they correspond to, not their own counter. A user manual written when the app is at v14 is user-manual(14).html, even if it is the first time that document has been produced.
Dev Badge

During development, a small version indicator lives in the app UI. Visible but unobtrusive — small, low contrast, in a corner that does not interfere with the interface. Removed before customer delivery.

/* Dev version badge — remove before customer delivery */
.dev-version-badge {
  position: fixed;
  bottom: max(6px, env(safe-area-inset-bottom, 6px));
  right: 8px;
  font-family: monospace;
  font-size: 10px;
  color: rgba(0, 0, 0, 0.22);
  pointer-events: none;
  z-index: 999;
  letter-spacing: 0.5px;
}

Chapter 3
Bug Tracking

Bugs get documented. Not in a separate tool, not in a mental list — in a consistent format that makes it possible to look back and understand what broke, when, why, and how it was resolved. This record feeds the development history.

Protocol
  • Every bug gets a version stamp. The version where the bug was discovered is part of the record. This makes it possible to trace when something was introduced.
  • Bugs are described in three parts: what the symptom was, what caused it, and how it was fixed. All three matter. The cause is often the most instructive part.
  • Bugs that cannot be fixed immediately are flagged as open. An open bug with no fix is still documented — it belongs in the technical reference under Known Limitations until it is resolved.
  • Patterns matter. If the same class of bug recurs (e.g., off-by-one errors in price calculations, state not resetting on form clear), that pattern belongs in the dev history and may warrant a new rule in the playbook.
For the AI When a bug is discovered and fixed in the course of a build session, note it explicitly. Format: [v##] [Symptom] — Cause: [what caused it]. Fix: [how it was resolved]. This note goes into the dev history entry at the ~20 version mark.
Bug Record Format
[v##] Short description of the symptom

  Cause:   What caused the bug. Be specific — "a logic error" is not
           useful. "The pork fat cost was being added even when
           porkFat was null rather than false" is useful.

  Fix:     What was changed to resolve it. Reference the function
           or section of code if it helps.

  Status:  Fixed in v## | Open — [reason not yet fixed]
Chapter 4
Delivery & Handoff

A deliverable to a customer is not the same as a working development build. Before anything goes to a real user, it passes through a defined checklist. The goal is that a customer receives something that is clean, professional, and free of development scaffolding.

Pre-Delivery Checklist
  • Dev version badge removed from the UI.
  • All console.log statements and debug output removed from the code.
  • No commented-out code blocks left in the file.
  • All placeholder text replaced — no [brackets], no "lorem ipsum", no sample data left in.
  • Default credentials changed or removed — no PIN: 1111 shipped as-is.
  • All form fields validated — no way to submit incomplete or malformed data.
  • Error states handled — every failure mode shows a useful message, not a blank screen or a raw error.
  • Tested on the actual target device and browser, not just a desktop simulator.
  • File name reflects the current version number and the project name.
  • If any data persists between sessions, the customer has been told where it lives and what "clearing it" means.
What Gets Delivered

The default deliverable is a single self-contained HTML file. No build step, no server required, no dependencies that need to be installed. The customer opens it in a browser and it works.

If the project requires more than a single file, a clear README-equivalent is included that explains what each file is and how to use it.

On self-containment External fonts and libraries loaded from CDNs are a dependency. If the customer opens the file without internet access, they should still get a functional, readable app. Where possible, fall back gracefully — system fonts instead of web fonts, for example. If a CDN dependency is unavoidable, document it explicitly.
Chapter 5
Clean Code Standards

The principles here are drawn from Robert C. Martin's Clean Code, adapted for language-agnostic use. The governing idea: code is written for humans to read, and only incidentally for machines to execute. A codebase that a stranger — or a future version of yourself — can read, understand, and safely modify without asking anyone questions is the measure of quality. Maintainability is not a bonus. It is the goal.

The compiler does not care whether your variable is named x or customersWithOverdueBalances. Anyone who touches this code after you cares enormously. Write for them.
Naming
Principle 1 Use Full Words. No Abbreviations.

Names should say exactly what a thing is. Abbreviations trade a few keystrokes for permanent confusion. A reader should never have to stop and decode a name.

Avoid
let usr = getUsr();
let amt = calcAmt(ord);
function updBal(acct, chg) { }
Prefer
let user = getUser();
let amount = calculateAmount(order);
function updateBalance(account, charge) { }
Principle 2 Names Reveal Intent.

A name should answer: what is it, why does it exist, and how is it used. If a name requires a comment to explain it, the name is not good enough.

Avoid
let d;           // days since start
let flag = true; // whether to show error
Prefer
let daysSinceStart;
let shouldShowErrorMessage = true;
Principle 3 One Word Per Concept, Used Consistently.

Pick one word for one concept and use it everywhere. Do not mix get, fetch, and retrieve interchangeably. Do not use customer in one place and client in another for the same thing. Consistency lets a reader build a mental model that stays reliable.

Principle 4 No Noise Words.

customerInfo, customerData, and customerObject are meaningless distinctions. info, data, and object add no meaning — they are noise. If two things need different names, give them names that reflect their actual difference.

Functions
Principle 5 Functions Do One Thing.

A function should do exactly one thing, do it well, and do it only. If you can describe what it does using the word "and," it is doing more than one thing. Split it.

Avoid — does four things
function processOrder(order) {
  validate(order);
  calculateTax(order);
  saveToDatabase(order);
  sendEmail(order);
}
Prefer — orchestrates named single-purpose functions
function submitOrder(order) {
  validateOrder(order);
  applyTax(order);
  saveOrder(order);
  notifyCustomer(order);
}
Principle 6 Function Names Are Verbs That Say What They Do.

Functions perform actions, so their names start with a verb. Specific enough that a reader knows what the function does without opening it.

Avoid
function data() { }
function run() { }
function handler() { }
Prefer
function loadOrderHistory() { }
function calculateTotalWithTax() { }
function handleLoginFailure() { }
Principle 7 Keep Functions Small. Prefer Few Arguments.

The ideal number of arguments is zero or one. Three or more is a strong signal that a redesign is needed — group related arguments into a named object.

Avoid
function createUser(
  firstName, lastName, email,
  phone, address, role
) { }
Prefer
function createUser(userDetails) {
  // userDetails.firstName
  // userDetails.email, etc.
}
Principle 8 No Hidden Side Effects.

A function that promises to do one thing should not secretly do something else. If a side effect is necessary, the function's name must make it explicit. Hidden side effects are among the hardest bugs to find.

Comments

The goal is code that explains itself. A comment is a failure of expression — a sign that the code could not communicate its own meaning clearly enough. Some comments are genuinely valuable. Many are noise or, worse, lies (code changes, comments don't).

Principle 9 Comments Explain Why. Code Explains What.

If a comment restates what the code is visibly doing, delete it. If a comment explains why a decision was made — a business rule, a known edge case, a deliberate trade-off — keep it. That context cannot be expressed in code.

Avoid — restates the obvious
// Increment the counter
count = count + 1;

// Check if user is active
if (user.isActive) { }
Prefer — explains the why
// Flat $165 up to 160 lbs, then $1/lb over.
// Rick confirmed this applies to all animals.
const baseRate = weight <= FLAT_RATE_CEILING_LBS
  ? FLAT_BASE_RATE
  : FLAT_BASE_RATE + (weight - FLAT_RATE_CEILING_LBS);
Principle 10 Never Leave Commented-Out Code.

Commented-out code is noise. It suggests it might matter when it doesn't, and rots as the surrounding code changes. Delete it. Version history preserves the past — commented-out code does not need to be a safety net in the source file.

Structure & Formatting
Principle 11 Use Constants for Magic Numbers and Strings.

A raw number sitting in code with no explanation is a magic value — the reader has to guess what it means. Named constants make the meaning explicit and make future changes safe.

Avoid
if (weight <= 160) {
  total = 165;
} else {
  total = 165 + (weight - 160);
}
Prefer
const FLAT_RATE_CEILING_LBS = 160;
const BASE_PROCESSING_RATE  = 165;
const OVERAGE_RATE_PER_LB   = 1;

if (weight <= FLAT_RATE_CEILING_LBS) {
  total = BASE_PROCESSING_RATE;
} else {
  total = BASE_PROCESSING_RATE
    + (weight - FLAT_RATE_CEILING_LBS)
    * OVERAGE_RATE_PER_LB;
}
Principle 12 Organize Code Like a Newspaper. High Level First.

Important, high-level things come first. Details and implementation go lower. A reader should understand the overall shape of a file by reading the top. Functions that call other functions appear before the functions they call.

Principle 13 Consistent Formatting Throughout.

Indentation, spacing, bracket placement — consistent throughout the entire codebase, always. Inconsistent formatting is distracting in the same way inconsistent spelling is distracting: the reader's brain spends cycles on noise instead of meaning.

Chapter 6
Mobile & Responsive Standards

Most apps built under this playbook are used on phones. Mobile is not an afterthought or a special case — it is the primary target. These standards apply to every app deliverable unless the app is explicitly desktop-only.

Touch Targets
  • Minimum touch target size: 44×44px. This is Apple's HIG recommendation and applies to any interactive element — buttons, toggles, links, inputs. Smaller targets cause errors and frustration.
  • Adequate spacing between targets. Two tappable elements should not be close enough that a normal thumb can accidentally hit both. Minimum 8px gap between adjacent targets.
  • No hover-only interactions. Touch screens have no hover state. Any information or control that only appears on hover is invisible on mobile.
  • Suppress tap highlight where custom feedback is provided. Use -webkit-tap-highlight-color: transparent when the element has its own active/pressed state. Otherwise leave the default.
Typography
  • Minimum body font size: 16px. iOS will auto-zoom input fields if the font size is below 16px. This breaks layout and is disorienting. Set font-size: 16px on all <input> and <textarea> elements.
  • Set inputmode on numeric inputs. inputmode="decimal" for numbers with decimals, inputmode="numeric" for integers. This presents the right keyboard without forcing type="number", which has undesirable behavior on some platforms.
  • Line length. On small screens, full-width text can be hard to read. Max content width of 680–720px for prose. Below that, let it be full-width.
Layout
  • Respect safe areas. iPhone notches and home indicator bars intrude on the screen. Use env(safe-area-inset-bottom) and env(safe-area-inset-top) for fixed elements near screen edges. Include the viewport meta tag: viewport-fit=cover.
  • Fixed bottom elements need bottom padding. Any fixed bar at the bottom of the screen — a submit button, a nav bar — needs padding-bottom: max(12px, env(safe-area-inset-bottom)) to clear the home indicator.
  • Prevent unwanted zoom on double-tap. Set touch-action: manipulation on interactive elements where double-tap zoom would be disruptive.
  • Disable user-scalable only when justified. user-scalable=no in the viewport meta should not be used casually — it breaks accessibility for users who need to zoom. Only use it when the layout is genuinely broken by pinch-zoom and there is no better solution.
Required viewport tag <meta name="viewport" content="width=device-width, initial-scale=1.0, maximum-scale=1.0">

For apps targeting home screen installation: also add <meta name="apple-mobile-web-app-capable" content="yes">
Mobile Performance
  • No layout thrash. Reading then writing DOM properties in a loop forces repeated reflows. Batch reads together, then writes together.
  • Transitions on transform and opacity only. Animating width, height, top, or margin triggers layout recalculation on every frame. transform and opacity animate on the GPU and do not.
  • Debounce expensive event handlers. Scroll, resize, and input events can fire dozens of times per second. Handlers that do heavy work should be debounced.
Chapter 7
Accessibility Baseline

Accessibility is not a compliance exercise — it is building things that work for more people. The standards here represent the minimum floor. They apply to every deliverable.

Color & Contrast
  • Minimum contrast ratio: 4.5:1 for normal text, 3:1 for large text (18px+ regular or 14px+ bold). This is WCAG AA, the standard minimum.
  • Do not use color as the only signal. If an error state is shown only by turning something red, a colorblind user gets no information. Pair color with a label, icon, or message.
  • Test with the screen in grayscale. If the interface still communicates clearly in grayscale, color is being used as enhancement, not as a crutch.
Semantic HTML
  • Use the right element for the job. Buttons that do things are <button>. Links that navigate are <a>. Do not make a <div> clickable when a <button> is what it is.
  • Every form input has a label. Either a visible <label> element associated via for/id, or an aria-label attribute. Placeholder text is not a label — it disappears when the user starts typing.
  • Heading hierarchy is meaningful. <h1> through <h6> should reflect actual document structure, not be chosen for their default visual size.
  • Images have alt text. Decorative images use alt="". Meaningful images describe what they show.
Interaction
  • Keyboard navigability. Every interactive element should be reachable and operable by keyboard alone. Tab order should follow the visual order of the page.
  • Visible focus indicators. The browser's default focus ring should not be suppressed without providing a replacement. A user who navigates by keyboard needs to know where focus is.
  • Error messages are descriptive. "Error" is not a message. "Please enter a customer name before saving" is a message. Errors should tell the user what went wrong and what to do about it.
  • Do not auto-advance focus unexpectedly. Automatically moving focus when a user makes a selection (e.g., advancing to the next field mid-PIN entry) can disorient screen reader users. Use it deliberately and sparingly.
Chapter 8
Security Basics

Most of the apps built under this playbook are small, local, single-file tools — not high-value attack targets. But basic security hygiene costs nothing and prevents a class of problems that are embarrassing and sometimes serious.

What Goes in localStorage

localStorage is visible to any JavaScript running on the page, persists indefinitely, and is not encrypted. It is appropriate for non-sensitive user preferences and locally-generated app data. It is not appropriate for the following:

  • Passwords or PINs in plaintext
  • Authentication tokens or session credentials
  • Social Security numbers, financial account numbers, or government ID numbers
  • Medical information
  • Any data the user would consider private and expect to be protected
On PINs specifically A PIN stored in localStorage in plaintext is readable by anyone who opens DevTools. For apps where the PIN is low-stakes (it protects a local order form, not a bank account), this may be an acceptable trade-off — but the limitation should be documented and the customer should understand it. Never store PINs in plaintext for anything with real security requirements.
Input Handling
  • Escape user-generated content before rendering it as HTML. If a user types a name and that name gets inserted into the DOM via innerHTML, a malicious user can inject arbitrary HTML or script. Use a sanitizing function or textContent instead of innerHTML wherever user input is displayed.
  • Validate on submission, not only on input. Client-side validation that only fires on the input event can be bypassed. Always validate again at the point of saving or submitting.
  • Do not trust the URL. If the app reads parameters from the URL (query strings, hash values), treat them as untrusted input — validate and sanitize before using them.
/* Escape user content before inserting into the DOM.
   Use this instead of innerHTML when displaying
   any value that came from user input. */
function escapeHtmlCharacters(rawText) {
  return String(rawText)
    .replace(/&/g, '&amp;')
    .replace(/</g, '&lt;')
    .replace(/>/g, '&gt;')
    .replace(/"/g, '&quot;');
}
Sensitive Data in Code
  • No API keys, passwords, or secrets in source code. A single-file HTML app delivered to a customer is readable source. Any secret embedded in it is exposed.
  • No real customer data in example or default values. Placeholder data should be obviously fictional. Do not use real names, real phone numbers, or real records as defaults or examples.
  • Default credentials must be changed before delivery. Any app with a login — even a simple PIN — ships with either no default credentials or with an explicit first-run setup flow that forces the user to set their own.
Chapter 9
Performance Baseline

"Fast enough" means the app responds to a user action in under 100ms. At 100ms, a response feels instantaneous. At 300ms, it feels slightly slow. At 1000ms, users start to wonder if something is broken. These are not strict rules — they are calibration points.

Baseline Expectations
  • First render is immediate. The app should display something useful within one second of the file being opened, on a mid-range phone, on a slow connection. If it takes longer, investigate why.
  • Interactions are synchronous where possible. Button presses, toggles, and form field updates should respond visually before any async operation completes. Show state change immediately, then do the work.
  • Scrolling is never janky. If scroll performance is poor, the most common causes are: expensive operations in scroll handlers, heavy DOM trees, and non-composited CSS animations. Address those in order.
Common Mistakes to Avoid
MistakeWhy It HurtsFix
Rebuilding the entire DOM on every state change Re-rendering a list of 200 items when one changes is 200× more work than it needs to be Update only the elements that changed
Blocking the main thread with synchronous work Any synchronous operation over ~50ms will cause a visible freeze Break heavy work into chunks, or use async patterns
Loading unused code Every byte of JS must be parsed before anything runs Only include what the app actually uses
Animating layout properties Width, height, margin, top trigger reflow on every frame Animate transform and opacity only
No loading feedback on async operations The user doesn't know if something is happening or broken Show a visible state change immediately; resolve it when done
Chapter 10
Development History Protocol

The development history is a running record of what happened to a project and why. It is not a changelog — it is a narrative that captures what was built, what broke, and the reasoning behind the decisions that shaped the project. Someone reading it should be able to understand how the project got to where it is without reading all the code.

Protocol

A new development history entry is written approximately every 20 versions. The exact trigger is judgment-based — if a lot happened in 15 versions, write it. If not much changed in 25, wait. The goal is meaningful chunks of work, not mechanical scheduling.

What to record
  • Features added — what was built and what problem it solved.
  • Bugs found and fixed — what broke, what caused it, how it was resolved. Include bugs that were instructive.
  • Architectural decisions — why was something built a particular way? What alternatives were considered?
  • Things removed or changed — what was undone and why.
  • Open questions — deferred decisions, known limitations, unresolved issues.

The file is named dev-history(N).html where N is the current app version. Each new entry is prepended — most recent first. The file is cumulative. Old entries are never deleted.

Write while it is fresh A history written weeks after the fact misses the decisions and reasoning that seemed obvious at the time. The most valuable part of a development history is the "why" — and that fades quickly.
Entry Template
Versions [N] – [N+~19] · [Month Year]
[Short descriptive title for this phase of development]
Features Added
  • [Feature name] — [what it does and why it was added]
Bugs Found & Fixed
  • [v##] [Symptom] — Cause: [what caused it]. Fix: [how it was resolved].
Architectural Decisions
  • [Decision] — [reasoning, alternatives considered, constraints that drove the choice]
Things Changed or Removed
  • [What changed] — [why]
Open Questions & Known Limitations
  • [Issue or question] — [current status or plan]
Chapter 11
Technical Reference Protocol

The technical reference is the authoritative document for how a project is built — its architecture, data structures, constants, functions, storage, and external dependencies. Its intended reader is a developer who needs to understand the project precisely enough to modify it safely.

Not every project needs a deep technical reference from day one. A small, simple project may only need one eventually. A growing project should start one early, while architectural decisions are still fresh.

Scope — What to Include
SectionContentsInclude When
Architecture Overview What the system is, how it is organized, what the major pieces are and how they relate. Always
Constants & Configuration All named constants, their values, units, and meaning. Pricing, limits, defaults. Always
Data Structures The shape of the primary objects the app works with. Fields, types, valid values. Always
Storage Where and how data is persisted. Keys, formats, what gets saved, what gets cleared. When data is stored
Function Index Significant functions — name, purpose, inputs, outputs, side effects. When codebase exceeds ~20 functions
External Dependencies Libraries, APIs, fonts, or services the project cannot function without. When any exist
Known Bugs & Limitations Current known issues not yet fixed, with trigger conditions and status. When any exist
Keep it current An out-of-date technical reference is worse than none — it actively misleads. Update relevant sections every time a meaningful architectural change is made. At minimum, review it when writing each dev history entry.
