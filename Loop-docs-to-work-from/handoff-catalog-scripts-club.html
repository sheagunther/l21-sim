Read This First

This document hands off the design work done in a single session on March 28, 2026. Three interconnected features were designed: a Shared Catalog Engine (refactoring Files and Scripts to share storage/UI code), the Scripts Box (a new sidebar section for managing skins and challenge scripts), and the Operators Club (a physical membership club with a cryptographic completion code challenge).

No code has been written. The current build is v.308. The output of this session is four documents (including this one) and a set of decisions. Your job is to read the reference docs, confirm with Shea which decisions still hold, resolve the pending questions, and start building Phase 0.

Before You Write Any Code
Read the Coding Standards document and the Dev Playbook — they are Project files. Then read the Unified Implementation Plan. Then read this document's Section 7 (Codebase Orientation) and Section 8 (Traps and Gotchas). Then talk to Shea about the pending decisions in Section 4. Only then should you start on Phase 0.
Section 1
The Situation

Shea designed the Scripts Box feature and wrote a detailed technical spec (scripts-box-spec.html). During this session, we worked through several design issues that the spec raised:

  1. The skin refactor problem. The Scripts Box needs to display readable skin source text in an editor, but the current skin system has no source text — skins are hardcoded JS objects. Each foundational skin must be rewritten as an authored source string that gets parsed into the runtime object at init. Same treatment as CBX challenge definitions.
  2. The Files/Scripts overlap. Shea noticed that Files and Scripts are the same pattern — named items in a localStorage catalog with a 16-item limit, naming rules, sort controls, and list UI. Rather than duplicate 250 lines of catalog management code, we designed a shared catalog engine that all four catalogs (Files, Skins, CBX Local, CBX Net) use. This also future-proofs FTP-for-scripts.
  3. The Operators Club. Shea wants a physical membership club. Operators solve a challenge, get a completion code, mail in an application with $16 in two-dollar bills, and receive a member number and postcard. We designed the code format (8 groups of 4 characters, 143-bit encoding), the stats philosophy (operator actions is the only published stat), and ten candidate challenges.

The work is phased into six steps (P0–P5), each independently testable. No code exists yet. Phase 0 (shared catalog engine) is the starting point.

Section 3
What's Been Decided

These decisions were made during the session and Shea has signed off on them. They should be treated as settled unless Shea explicitly changes his mind.

Architecture Decisions

Decided Shared catalog engine. Files and Scripts share one set of storage, naming, sorting, and list-rendering code. The engine is parameterized by config objects. All four catalogs (Files, Skins, CBX Local, CBX Net) use it.

Decided Skin source string architecture. Each foundational skin is rewritten as an authored source string constant. The runtime SKINS object is populated by parsing these strings at init. The source string is the authoritative representation. Same pattern as CBX challenge definitions.

Decided Six-phase implementation. P0 (catalog engine) → P1 (skin refactor + Scripts shell) → P2 (Skins sub-section) → P3 (CBX Local + Networked) → P4 (.l21s format + upload) → P5 (Operators Club Challenge). Strictly linear dependencies.

Decided Phase 0 before Phase 1. The Files refactor onto the shared engine is a separate deliverable that ships and gets tested before the skin refactor starts. Risk isolation: if the catalog refactor breaks Files, catch it before adding the skin variable.

Decided Challenge source strings. Built-in challenge definitions (9 local + 1 networked) must also be converted to source string constants so they can be displayed in the read-only editor. This happens in Phase 3, parallel to what Phase 1 does for skins.

Scripts Box Decisions

Decided Three sub-sections: Skins, CBX Local, CBX Networked. Positioned between Challenge and Buses in the sidebar.

Decided Collapse behavior. Scripts group starts collapsed. All three sub-sections start collapsed. Wired into Expand All / Collapse All buttons.

Decided Foundational scripts. Read-only, lock icon, always at top of file list, do not count against 16-slot limit. Operator can read and copy the source text but cannot modify it.

Decided Foundational skin count: 3 (OG, Winamp, Sunrise). Foundational CBX Local count: 9. Foundational CBX Networked count: 1 (Chain Sum). These will increase by 1 when the Operators Club Challenge is added in Phase 5.

Decided Old challenge custom section removed. The textarea, REGISTER button, and custom challenge list in the Challenge sidebar section are removed in Phase 3. Replaced by a wayfinding hint.

Operators Club Decisions

Decided Physical mail application. Printed form, $16 in two-dollar bills (8 bills), pre-addressed stamped postcard. Mailed to North Windham, Maine.

Decided Member numbers are random. Not sequential, not encoded. Same principle as variant numbers.

Decided Cool number auctions. Reserved numbers auctioned in dribs and drabs over time. Not all at once.

Decided Newsletter. Monthly email. Content at Shea's sole discretion. Not a product update list.

Decided Variant registry. Separate program, same mail infrastructure. $32 in two-dollar bills (16 bills). Does not require club membership.

Decided No leaderboards. Shea explicitly does not want to maintain leaderboards.

Decided Duplicate rule. If a completion code has already been used, the application is rejected and Shea spends the $16 on ice cream. The math guarantees duplicates are fraud, not coincidence.

Decided Challenge is re-rollable. Operators can start the challenge as many times as they want. Each start generates a new seed. No penalty for restarting.

Decided Knowing the solution doesn't help. The challenge is a recipe, not a puzzle. The procedure is printed in the description. You still have to do it by hand.

Decided Five-minute sizing. The challenge is sized so a skilled operator finishes in about five minutes. This is a design constraint on scope, not a time limit. No timer, no enforcement.

Stats Decisions

Decided Operator actions is the only published stat. Published alongside member number and date. No wall time, no tick count in the public record.

Decided All stats encoded in the code privately. The completion code encodes tick count, operator actions, wall time, and a hash. Shea sees all stats when verifying. Only operator actions gets published.

Decided Public record per member: three fields. Member number, date, operator actions. That's it.

Decided Completion code format: 8 groups of 4 characters, 22-character unambiguous alphabet, 143 bits total. 48-bit seed, 16-bit ticks, 12-bit actions, 12-bit wall seconds, 55-bit hash.

Decided Verification tool is a standalone HTML file. No server. No database. Paste code, get result.

Before Phase 3

It can only be made once. See the Operators Club Design Document, Section 4 for full candidate descriptions and comparison matrix.

Pending Challenge scoring participation. Does the club challenge participate in the normal scoring system (par, history), or is it a separate system that only produces a completion code?

Pending Visual distinction. Should the club challenge be visually distinct from other foundational challenges in the file list?

Pending Copy-to-clipboard. Should the completion code have a copy button, or be display-only to reinforce the analog nature?

Pending Code backup. Should the simulator store the most recent completion code in localStorage?

Non-code decisions (no phase dependency)

Pending Launch timing. Club at launch or post-launch?

Pending P.O. box or home address?

Section 5
Open Design Questions

These are implementation-level questions that can be resolved by the implementing Claude instance, ideally in consultation with Shea. They're listed here so you know they exist.

Design Shared vs per-catalog dialog elements. Should naming and delete dialogs be one shared DOM element repointed per catalog, or separate elements per catalog?

Design Per-catalog UI state storage. Module-level variables won't scale to four catalogs. State object per config, parallel state Map, or properties on the config?

Design Post-name-confirmation callback. Files arms a save pipeline; Scripts navigates to editor. Cleanest hook pattern for type-specific behavior after naming?

Design SKINS declaration. const SKINS = {} with properties set at init, or let SKINS? Coding standards favor const aliases.

Design CSS variables: single source of truth. The :root CSS and the skin source strings both define section colors. Recommendation from the session: source strings are authoritative, CSS is a pre-JS-execution cache. Both must stay in sync manually.

Design Winamp/Sunrise overrides for new section colors. Hand-pick to match palette or derive by formula?

Design VALID_CSS_VARS set. Auto-generated from :root at runtime, or hardcoded?

Design Custom skin body class namespace. skin-my-theme or skin-custom-my-theme?

Design CBX dry run event loop yielding. 250ms synchronous block acceptable, or yield between dry runs?

Design Hash salt obscurity. Buried or clearly named?

Design 22 vs 24 character alphabet. Current design excludes 5/S. Including them gives more hash bits. Excluding reduces transcription errors.

Section 6
What to Build Next

Phase 0: Shared Catalog Engine. This is a pure refactor. Zero functional change to the operator.

Step 1: Read

Read the Coding Standards doc. Read the current Files implementation in v.308 (lines ~6146–6860). Understand every function, every state variable, every DOM element ID. The catalog engine must be a precise extraction — not an approximation, not a reimagining. The existing Files behavior is the contract.

Step 2: Design the config object shape

Resolve the three open design questions for Phase 0 (shared vs per-catalog dialogs, UI state storage, post-confirmation callback). Propose the config object shape to Shea before writing code. The config shape is the API contract for all four catalogs.

Step 3: Extract and refactor

Write the shared catalog engine functions. Refactor Files functions to be thin wrappers that call the engine with CATALOG_CONFIGS.files. Leave all Files-specific behavior (File Save pipeline, FTP, encode/decode) untouched.

Step 4: Test

Verify that every Files operation works identically to v.308: create, name, save (armed + batch write), load, delete, sort by name, sort by time, overwrite dialog, partial save, cancel save, FTP send, FTP receive, FTP accept/reject. Verify localStorage format is unchanged — existing saved files must survive the upgrade.

Step 5: Deliver

node --check. Update APP_VERSION (both filename and JS constant — see Traps section). Deliver as v.309 (or whatever the current next version is at the time).

Do Not Combine Phases
Phase 0 must ship alone. Do not combine it with Phase 1. The Files refactor and the skin refactor are two independent risks and must be tested independently. If P0 breaks something, you need to know before P1 adds noise.

Section 8
Traps and Gotchas

APP_VERSION drift

The #1 process failure on this project. Every build must update BOTH the filename (loop2-stage2 (NNN).html) AND const APP_VERSION in the JS (line ~6150). These must always match. The version has drifted three times in the project's history because the filename was updated but APP_VERSION wasn't. Search for APP_VERSION — there is exactly one instance.

Bus capability parity

Buses A, B, C, and D are identical in capability. Never describe them as different or lesser. This applies to documentation and to code comments.

Operators, not "Loopers" or "flowheads"

L21 users are called operators. This is the machine's functional term. Applies permanently across all documents, code comments, UI text, and conversation.

L21 is not anti-anything

Never frame L21 as a critique of or response to modern computing. The correct framing is curiosity, craft, joy, and a different vantage point on computation.

No unrequested features

Build exactly what is specified. Do not add UI elements, keyboard shortcuts, helper functions, or "nice to have" features that were not explicitly requested. This is listed as an anti-pattern in the Coding Standards.

node --check before every delivery

Syntax check with node --check is required before every delivery. Never deliver without a passing check.

Never deliver two files when one is needed

The simulator is a single HTML file. Always deliver one file.

Never stop mid-build to ask "continue"

Keep going automatically. Complete the build.

Finding bugs is Claude's job

In this vibe coding arrangement, finding bugs in code is the implementing Claude's responsibility. Do not frame bug discovery as something Shea "couldn't find."

The easter egg

There is a deliberately obfuscated easter egg in the codebase (identifiers like _v0, _c0, _k0, _p0, _ee0). Never reference what it does or how to trigger it unless Shea raises it first.

Scoring tension

Time-weighting in the scoring system penalizes exploratory learning. This is a known tension — worth tracking but not something to "fix" without Shea's direction.

Skin source strings and CSS :root must stay in sync

After Phase 1, the new section color variables (--sec-skins, etc.) will exist in two places: the CSS :root block and the Winamp/Sunrise skin source strings. If you change one, change the other. There is no automated sync.
