Architecture Constants

These values are fixed at the architecture level. They are baked into every timing calculation, buffer size, and protocol design in the system. They cannot be changed at runtime.

Core Constants

BUS_N24Bus pipeline length in slots. One bit per slot per tick.
INJ_N17Inject channel length. Equals BPW — holds exactly one word in flight at a time.

Bit array index 0 = write head, index 1 = gate, index 2 = read head (rIdx() = 2). A bit written at index 0 takes N−2 ticks to reach index 2, where N is the total bit array length.

Big Loop Inspection Pipeline — Eject Positions

Words pass through inspection components in this order on every Big Loop rotation. Each component is exactly 17 bits (one word) downstream of the previous.

ConstantValueComponent
PM1_EJECT_IDX444Pattern Matcher 1 — first inspection point
PM2_EJECT_IDX427Pattern Matcher 2 — 17 bits after PM1
TG1_EJECT_IDX410Threshold Gate 1 — 17 bits after PM2
TG2_EJECT_IDX393Threshold Gate 2 — 17 bits after TG1
Pipeline order
A word destroyed by PM1 in destructive mode never reaches PM2, TG1, or TG2. Inspection is sequential and irreversible within a single rotation.
Chapter 2
The Tick Model

Every tick executes exactly four steps, always in this order, without exception. Understanding the order resolves most timing-sensitive behaviors that would otherwise seem counterintuitive.

StepNameWhat happens
1READAll bus sources sample bits[2] (read head) of their source loop. ALU capture samples bits[2] of the ALU loop. Memory capture samples bits[2] of the Memory loop. PM and TG inspection fires. This happens before the gate and before rotation.
2GATEIf a loop's gate is closed (gateOpen = false), bits[1] is zeroed before rotation.
3ROTATEEach non-paused loop shifts its bit array rightward by one position. The bit at bits[N-1] wraps to bits[0].
4WRITEInjected bits, bus deliveries, and register writebacks are placed at bits[0] of their destination loops. Bus arrays shift, delivering bits toward their destinations. ALU result updates if any input register changed.
Critical subtlety
ALU capture (Step 1) is suppressed while any writeback pipeline is active (alu.wbPos >= 0). This prevents an outgoing register word from being immediately re-captured back into a register.
Bus trailing delivery
Turning a bus OFF does not immediately stop delivery. Up to BUS_N=24 bits already in the pipeline continue to deliver over the next 24 ticks. The live-bit counter (busXLiveBitCount) ensures in-flight bits complete delivery.

Equivalence
Buses C and D are identical in capability to A and B. No bus is lesser. The documentation never describes C or D as different or limited versions of A or B.
Memory — 16 Slots, 4-bit Addressing

The 4-bit address register (mem.addrBits) selects the active slot. Address evaluates to (b3<<3)|(b2<<2)|(b1<<1)|b0. Slots 0–15. In address-read mode (mem.addrRead), destination slot is decoded from the incoming word's top 4 bits (b15–b12).

Chapter 4
Code Naming Conventions

Loop 2.1 follows Clean Code principles enforced through a 116-identifier naming audit. These conventions must be preserved in all future work on the codebase.

Clean Code Principles
RuleDetail
Whole words onlyNo abbreviations in identifiers unless they are universally understood acronyms (ALU, PM, TG, CBX, P2P). readHeadBitIndex not rIdx in new code.
Descriptive function namesFunction names must describe what the function does. toggleWorkingScratchCapture() not toggleWSC(). Code that reads like prose.
No magic numbersAll architecture constants are named: BPW, BUS_N, INJ_N, PM1_EJECT_IDX, etc. Never write 17 or 41 or 444 directly.
One responsibility per functionFunctions do one thing. If a function name requires "and", split it.
State objects are explicitAll component state lives in a named object (alu, mem, CBX, netState). No scattered globals.
Comments explain why, not whatThe code describes what happens. Comments explain why the design choice was made.

Chapter 6
Loopscript 1.0 — .lps

Loopscript is a compact notation for describing Loop 2.1 algorithms at the level a trained operator thinks about them — not tick by tick, but the meaningful moves. File extension .lps. Version 1.0.

Document Structure

Every .lps file has three sections in order, separated by --- (three or more dashes).

TITLE:   Name of the algorithm
AUTHOR:  operator-handle
DATE:    yyyy-mm-dd
VARIANT: 2.1          // or a variant number e.g. 00003
IN:      description of inputs
OUT:     description of outputs
NOTES:   optional free text
---
SETUP
// component configuration lines (only what differs from default)
---
EXEC
// algorithm steps, one per line

Comments begin with // and run to end of line. They may appear anywhere. A comment always goes on the line immediately below the step it explains — never at the end of the same line as an action.

Silence means default. If SETUP does not mention TG2, TG2 is at its default state. Readers do not need to wonder.

Chapter 7
P2P Wire Protocol

All traffic on the WebRTC data channels (Buses F and G) is JSON. Every message is dispatched by its type field — unless it contains a cbxType field, in which case it is a CBX protocol message and is intercepted before any loop delivery.

Message Types — Raw P2P Channel
typeDirectionPayload fieldsPurpose
wordEither directionvalue: 0–65535Deliver a 16-bit data word. Queued into word buffer, serialized to bit queue, delivered to destination loop at one bit per tick.
identityEither directionhandle: string, machineId: 4-char stringSent immediately on channel open. Establishes peer name and machine ID for display and routing.
pingInitiator → Peert: timestamp (ms)Latency measurement request. Sent on 1-second interval.
pongPeer → Initiatort: original timestamp echoedLatency measurement response. Round-trip = now − t.
Interception order
Every incoming message is checked for a cbxType field first. If present, it is routed to cbxReceiveMessage() and the word-delivery path is skipped entirely. If not present, the type field dispatches to the appropriate handler. CBX messages are processed asynchronously on receipt — they are not queued for the next tick.
Networking messages
Named Networking messages also travel on the P2P data channel alongside word and CBX messages. They are identified by a netType field and dispatched to netReceiveMessage().
Two-Stage Word Delivery

Words arriving from P2P are not placed directly into a loop. They go through two stages:

Stage 1 — Word buffer: Received words are pushed into _p2pWordBufL or _p2pWordBufR (FIFO queue). Decouples asynchronous WebRTC delivery from the synchronous tick model.

Stage 2 — Bit queue → pipeline: On each tick, if the inbound bit queue is empty, the oldest word from the word buffer is serialized into the bit queue as a 17-bit sequence. The bit queue drains one bit per tick into the inbound bus pipeline, delivering the word to the destination loop exactly as if it had been sent from another loop via an internal bus.

Availability States — CBX_AVAILABILITY

Chain Sum — Phase Sequence

The participation state machine moves through these phases in order:

idle→ invited→ assigned→ waiting_for_sum→ staged→ released→ submitted→ done

Jeopardy lockout
If the operator presses RELEASE before the chain sum marker has arrived and been staged, a 3-second lockout fires. The RELEASE button is disabled for 3 seconds as a penalty. After lockout the button re-enables — the operator must wait for the marker.
The Chain Sum Marker

Running sum forwarding between machines uses a two-word raw P2P transmission rather than a JSON CBX message — because the sum is real data the operator receives and works with.

ConstantValueMeaning
CBX_CHAIN_SUM_MARKER0xCB00 (51968)Reserved marker word. "Challenge Bus, sum follows." Sent immediately before the running sum value.

When a machine forwards the running sum: it sends { type: 'word', value: 0xCB00 } followed immediately by { type: 'word', value: runningSum }.

The receiving machine's p2pQueueInboundWord calls cbxCheckIncomingChainWord() before queuing any word. When 0xCB00 arrives, the function sets CBX._incomingMarkerSide and consumes the word (not queued). The next word from the same side is treated as the incoming sum and also consumed.

Reserved value
0xCB00 (51968 decimal) is the one value in the 0–65535 range treated specially by the P2P channel. It will be silently consumed by the CBX interception layer. Challenges and operators sending data over P2P connections should avoid generating this specific value.

Hub Election Algorithm

When the watchdog fires (3 missed heartbeats), remaining members start a Bully election:

StepAction
1. Watchdog firesMember sets electionInProgress = true, broadcasts net_hub_elect with its own machine ID as candidate.
2. Receive candidateIf incoming candidate ID < own ID (lower wins), adopt that candidate and forward it. If own ID is lower, broadcast own ID.
3. Claim timeoutAfter 2 seconds without a better candidate, the lowest-ID machine broadcasts net_hub_claim and assumes hub role.
4. On claim receivedAll members update their net.hubId, clear election state, reset watchdog timer.
Bully variant
Lowest machine ID wins (not highest). Machine IDs are 4-character strings from the charset ABCDEFGHJKLMNPQRSTUVWXYZ23456789. String comparison is used — 'AAAA' < 'BBBB'.
