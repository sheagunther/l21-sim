Introduction
What This Machine Is

Loop 2.1 is a manual flow computer — a computing paradigm where the human operator makes every routing and operational decision in real time. There is no stored program, no fetch-decode-execute cycle, no instruction pointer. Data circulates continuously in hardware loops and the operator's job is to intercept it, transform it, and route it to the right place at the right time.

This is not a simulation of a conventional computer. It is a different model of computation, one that makes the flow of data visible and physical. Every word of data is 17 bits: one marker bit followed by 16 data bits representing an unsigned integer from 0 to 65,535. Words circulate clockwise in their loops at whatever clock rate the operator sets. The operator sees them pass the read head as binary and hexadecimal values, captures them into the ALU, operates on them, and reinjects results.

The Core Paradigm
The machine runs continuously. You don't tell it what to do — you intercept what's happening and shape it. Think of it less like programming and more like operating a mixing board or a patch bay. Timing matters. Position matters. Every action you take is an operator action, and the scoring system counts them.

The machine runs in a single HTML file with no dependencies, no server, and no installation. Open it in a browser and it runs. Everything is visible — right-click and View Source in any browser to read the full implementation, or open the file in a text editor. Source code, state, and session log are all accessible and nothing is hidden behind abstraction.

Architecture
Four Loops, One ALU, Sixteen Memory Slots

The machine has four circular bit-shift registers of different sizes, a single ALU with four registers, sixteen addressable memory slots, a pattern matcher, a threshold gate, and a set of buses for routing data between components. All components operate synchronously on the master clock.

Working Loop
12 words · 204 bits
The primary workspace. Data enters here from the injection channel. Most operations begin and end here.
ALU Loop
16 words · 272 bits
Feeds the arithmetic unit. Words passing the read head are automatically captured into Register A or B.
Memory Loop
16 words · 272 bits
Connects to the 16 memory slots. Words are written to and read from addressed slots as they pass.
Big Loop
32 words · 544 bits
High-capacity staging. The Pattern Matcher and Threshold Gate operate here. Used for multi-word pipelines.

Each loop has three functional positions: the read head (position 2 from start, reads the passing word), the write head (position 0, injects new words), and the gate (position 1, can hold a word in place or let it pass). The marker bit identifies the first bit of each 17-bit word, letting the hardware track word boundaries as bits shift through.

The ALU has four 16-bit registers: A and B are inputs, C is the result, D is a working scratchpad. A comparator runs continuously on A and B. Supported operations include ADD, SUB, MUL (lower 16 bits), AND, OR, XOR, NOT, SHL, SHR, INC, DEC, NEG, and SWAP A↔B.

Buses
Routing Data Between Components

Buses are one-way data paths that carry words from a source to a destination. Each bus has a configurable source, destination, and on/off state. Buses A through D are the four internal routing buses. Each can connect any loop or component to any other.

A General routing — connects any source to any destination. Used for most cross-component transfers.
B General routing — identical capability to Bus A. Run two transfers simultaneously.
C General routing — identical capability to A and B.
D General routing — identical capability to A, B, and C.
E Challenge Bus — bidirectional interface to the Challenge Box. Inbound delivers challenge values; outbound carries your answers.
F P2P Left — connects to the machine to your left via WebRTC. Words exit left, words arrive right.
G P2P Right — connects to the machine to your right. Bidirectional, same architecture as Bus F.
H Challenge Network Bus — CBX protocol coordination layer. Carries chain challenge control traffic between Challenge Boxes.
Buses C and D
Buses C and D have identical capabilities to A and B. Any source and any destination is available on all four internal buses. There is no hierarchy.
Before You Begin
Starting the Machine

The machine opens in a paused state. Nothing moves until you start the clock. This is intentional — it gives you time to configure switches and buses before any data starts flowing.

  1. Open the HTML file in any modern browser. No server, no install, no dependencies. The simulator renders immediately.
  2. Find the Clock controls in the top-left of the main canvas area — a START button and a speed slider. Click ▶ START to start the clock. The button changes to ⏹ STOP and the tick counter begins incrementing.
    The machine is now running. Loops are empty so nothing interesting is happening yet — that changes the moment you inject a value.
  3. Set the clock speed using the slider. The recommended starting speed is 24 Hz — about two Working Loop rotations per second. This is fast enough to feel live and slow enough to watch individual words travel. You can adjust at any time, including while a challenge is active.
    Speeds above 60 Hz are supported. The machine fires multiple ticks per animation frame to keep up. Very high speeds (200+ Hz) are useful for running challenges quickly but make it hard to observe individual transfers.
  4. To pause at any time, click ⏹ STOP. The machine freezes mid-tick. All loop contents, register values, and state are preserved exactly. Click ▶ START to resume from the same point.
The Tick Counter
The tick counter in the header is the machine's clock. Every tutorial references ticks — "the word reaches the read head after 12 ticks," "your score is based on ticks elapsed." Keep an eye on it. It only increments when the machine is running.

The right sidebar contains all controls: Challenges, Challenge Network, P2P Bus Config, P2P Connection, Memory, Pattern Matcher, Threshold Gate, and more. Sections are collapsible. If you can't find a control, it's probably in a collapsed section. The main canvas shows the four loops, the bus strips, and the ALU and injection channel displays.

Tutorials
Eleven Operations from First Principles
Tutorial 01 Load a Value and Read It Back

The most fundamental operation: set a value on the input switches, inject it into the Working Loop, and watch it circulate. This establishes the inject→loop→read-head cycle that underlies everything else.

  1. Find the 16 input switches in the left sidebar, under the Inject section. Each switch is one bit, MSB at left. Try 0x00FF (lower 8 bits on). The hex display below the switches updates as you flip them.
    For quick testing, the RNG button next to the switches loads a random value instantly. Use it whenever you need a test value without caring what it is.
  2. Click SEND. A 17-bit word — marker bit followed by your 16 data bits — enters the Working Loop at the write head position.
    The inject LED flashes. The word takes one full loop rotation (12 clock ticks) to reach the read head.
  3. Watch the binary display at the Working Loop read head. The marker bit passes first as a bright dot, followed by the 16 data bits. The hex value appears in the readout to the right.
  4. The word continues circulating indefinitely — it will pass the read head every 12 ticks. Nothing destroys a word unless you actively route it out or overwrite it.
Word Position
The marker bit travels with the word as it circulates. Watch for the brighter dot at the start of each word passing the read head — that is the marker bit, and it is how the hardware tracks word boundaries as bits shift through.
Tutorial 02 ALU Arithmetic: Add Two Numbers

The ALU reads words from its loop automatically. Route a word there, configure an operation, and the result is available in Register C immediately.

  1. Inject your first operand (e.g. 0x0014 = 20) into the Working Loop using the switches and SEND.
  2. Turn on Bus A. Set its source to Working Loop and its destination to ALU Loop. The word transfers on the next tick the read head passes it.
    Watch the ALU Loop — your value should appear there within one Working Loop rotation.
  3. Turn Bus A off. Set the switches to your second operand (e.g. 0x000A = 10) and inject it into Working. Route it to ALU the same way.
    Bus A must be off before injecting the second operand, otherwise both values may enter the ALU Loop simultaneously — the second word would push the first past the read head before it loads into Register A, breaking the auto-capture sequence.
  4. In the ALU panel, the Comparator section shows Register A and B. Verify both values loaded. Select operation ADD. Register C shows the result immediately: 0x001E = 30.
  5. To use the result: set Bus A source to ALU Result (C), destination to Working Loop. The result word enters Working on the next transfer.
Register Auto-Capture
Words entering the ALU Loop are captured into Register A automatically on the first pass, then into Register B on the second pass. You do not need to manually load registers — just route two words into the ALU Loop in sequence.

The full operation set includes ADD, SUB, MUL (lower 16 bits), DIV, MOD, AND, OR, XOR, NOT, SHL, SHR (logical and arithmetic), INC, DEC, NEG, ABS, and SWAP A↔B. The Comparator continuously computes A compared to B, setting EQ, GT, LT flags readable in the flag display.

Tutorial 03 Memory Slots: Store and Retrieve

The sixteen memory slots are addressed by a 4-bit address register, which the operator sets using four toggle levers in the Memory section. Values are written and read as words pass through the Memory Loop.

  1. Set the 4-bit address levers to your target slot (0–15). The address display shows the decimal and hex equivalent.
  2. Route a value into the Memory Loop via Bus A (source: Working, destination: Memory Loop).
  3. Enable Write mode in the Memory panel. As the word passes the write position in the Memory Loop, it is captured and stored in the addressed slot. The slot display updates immediately.
    Write mode is destructive — the word is consumed from the loop when written. Disable Write mode once the value is stored, or the next word to pass will overwrite the same slot.
  4. Disable Write mode. Change the address levers to a different slot. Enable Read mode — the value in the addressed slot is injected as a new word into the Memory Loop on the next tick.
    Disable Read mode after one pass, otherwise the same slot will be re-injected every loop rotation, filling the Memory Loop with duplicate copies.
  5. Route the retrieved word back to Working via Bus A (source: Memory Loop, destination: Working Loop).
16 Slots × 16 Bits
Each slot holds one 16-bit value. All 16 slots are visible simultaneously in the memory grid display. You can use memory as a scratchpad, a lookup table, an accumulator set, or a register file — the machine assigns no meaning to slots. The CLEAR ALL button zeros all 16 slots at once. Per-loop word counters in the sidebar track how many words have passed each read head — directly analogous to hardware performance counters.
Tutorial 04 Route Between Loops: Copy and Move

Buses are the machine's wiring. You can run up to four simultaneous transfers using Buses A, B, C, and D. Each bus transfers one word per word-boundary crossing of its source.

  1. Inject three values into the Working Loop in sequence using the switches and SEND three times. You now have three words circulating.
  2. Set Bus A: source = Working, destination = Big Loop. Turn Bus A on. Words transfer one at a time as they cross the Working read head. Watch the Big Loop fill.
  3. To copy (not move) — leave Bus A running and the words continue circulating in Working too, since exiting via a bus doesn't destroy the word by default. To move — turn Bus A off after transfer and the word stays in the Big Loop only.
    Each loop has a gate at position 1 (between write head and read head). When enabled, the gate holds passing words rather than letting them continue. Combined with the eject function, the gate can destructively remove a specific word from a loop. Gate controls are in the left sidebar under each loop's section.
  4. Try running Bus A and Bus B simultaneously: A from Working→ALU, B from Memory→Big. Both transfers happen in parallel, one word per tick per bus.

You can route between any combination: Working↔ALU, Working↔Memory, Working↔Big, ALU↔Memory, ALU↔Big, Memory↔Big, and any loop to/from the P2P buses or the Challenge Bus E. Source and destination selectors are in each bus control panel.

Tutorial 05 Pattern Matcher: Filter and Transform

The Pattern Matcher (PM1) operates on the Big Loop. It inspects passing words against a configurable mask and match pattern, and either ejects matching words or rewrites them in place. A second, cascadable Pattern Matcher (PM2) operates in series.

  1. Route several values into the Big Loop. Set Bus A source to Working Loop, destination to Big Loop, and turn it on. Inject a mix of even and odd numbers — for instance, 4, 7, 12, 3, 8, 5 — using the switches and SEND. Turn Bus A off once all six are in the Big Loop.
    The Pattern Matcher and Threshold Gate only operate on the Big Loop. Values must be there before either component can act on them.
  2. In the PM1 panel, set the mask to 0x0001 (bit 0 only) by flipping the rightmost mask lever on. Set the pattern to 0x0000 (match when bit 0 = 0, i.e. even numbers).
  3. Enable PM1 in Copy mode (non-destructive). Matching words — the even numbers — will be ejected to the PM1 eject output. Set Bus A source to PM1 eject and destination to Working to receive them.
  4. For destructive filtering, switch to Destructive mode. Matching words are removed from the Big Loop and ejected. Non-matching words continue circulating.
  5. Enable Rewrite Mode on PM1. Set a change mask and change value. Matching words have specific bits rewritten in place rather than being ejected. Useful for flag manipulation and in-loop transformation.
PM2 Cascade
PM2 can receive PM1's matched output directly via cascade mode, applying a second filter to what PM1 already matched. This creates a two-stage filter pipeline within the Big Loop — match a broad set with PM1, refine with PM2 — without any operator routing between stages.
Tutorial 06 Threshold Gate: Conditional Ejection

The Threshold Gate (TG1, and a second TG2 in cascade) monitors words in the Big Loop and ejects them when their value crosses a configurable threshold. Unlike the Pattern Matcher which compares bits, the Threshold Gate compares numeric magnitude.

  1. Route a stream of values into the Big Loop — a mix of values above and below 1000 works well as a test.
  2. In the TG1 panel, set the threshold value using the levers or by typing directly into the value field. Set it to 1000.
  3. Select a comparison mode: ≥ threshold ejects values 1000 and above; ≤ threshold ejects values 1000 and below; = threshold ejects exact matches only.
  4. Enable TG1. Words matching the condition are ejected from the Big Loop. Route TG1's output to Working via Bus A to collect them.
  5. Enable Clamp Mode on TG1. Instead of ejecting matching words, TG1 rewrites them in place to the clamp value. Words above 1000 become exactly 1000 — a ceiling clamp. Reverse the condition for a floor clamp.
TG → PM Cascade
TG2 can receive TG1's output directly, creating a two-gate pipeline. A value must pass TG1's test before TG2 even sees it. Combined with PM1→PM2 cascade, the Big Loop can run a four-stage conditional pipeline: PM1 → PM2 → TG1 → TG2, all without operator routing between stages.
Tutorial 07 The Big Loop: Multi-Word Pipelines

The Big Loop's 32-word capacity makes it the natural staging area for operations that require holding multiple values simultaneously — sorting passes, running totals, buffered streams. This tutorial builds a simple running maximum finder.

  1. Inject 8 random values into the Big Loop using the switches and SEND eight times. Confirm all 8 words are circulating.
  2. Route the first word from Big into the ALU Loop via Bus A. It loads into Register A. Note the value.
  3. Route the second word from Big into the ALU Loop. It loads into Register B. Set the ALU operation to MAX (or use the comparator: if A < B, swap A↔B first). Register C now holds the larger of the two values.
  4. Route Register C (the result) back into the Big Loop — this is your current running maximum. Do not route the smaller value back. Repeat: pull the next word from Big into the ALU Loop (it loads into B, displacing the old B), compute MAX again, route C back. Each pass through the ALU replaces the running maximum if the new value is larger.
  5. After all 8 values have been compared, the remaining word in the ALU is your maximum. Route it to Working or read it from the ALU display.

This is a sequential scan in O(n) operator actions — the same algorithm a CPU runs, made visible one comparison at a time. The Big Loop's size means you can hold all values in flight simultaneously while you pull them off one at a time for comparison.

Tutorial 08 Bus E and the Challenge Box

Bus E is the interface to the Challenge Box — the machine's built-in problem-generator and scorer. Challenges deliver values to you via Bus E inbound and expect answers back via Bus E outbound. The Challenge Box knows the correct answer and scores your submission on ticks elapsed and operator actions taken.

  1. Open the Challenges section in the right sidebar. Select Add X Numbers from the type dropdown. Set the count to 3.
  2. Click ▶ NEW CHALLENGE. Bus E activates automatically — its source and destination are preconfigured for the challenge type, so you do not need to set them. The three challenge values begin queuing for delivery via Bus E inbound.
  3. Set Bus A source to Bus E inbound and destination to Working Loop. Turn Bus A on. The three challenge values enter your Working Loop one at a time as they arrive from Bus E.
  4. Route all three values through the ALU in sequence: Working → ALU, ADD, result back to Working. After two additions you have the running sum. After the third addition you have the total.
  5. Set Bus A source to Working Loop, destination to Bus E outbound. Route your sum word to Bus E. The Challenge Box receives it and evaluates — correct answers show your score, ticks used, and operations count.
Scoring Modes
Two scoring modes are available: Efficiency (ticks 55%, operator actions 45%) and Speed (adds real elapsed time at 35%, reducing ticks and ops weight). Switch before starting. Mid-challenge switching is locked to prevent gaming.

Available challenges: Add X Numbers, Multiply Two Numbers, Transform Every Nth Value, Filter (pass matching values), Sort Values, Find the Maximum, XOR Checksum, Count Matches, Accumulate to Threshold, and Chain Sum (multi-computer, covered in Tutorial 10).

Tutorial 09 P2P Connections: Buses F and G

Two machines can connect directly via WebRTC without any server infrastructure. Once connected, words flow between machines via Buses F (left) and G (right), exactly as they flow between loops internally. There is also a text chat channel on each connection.

  1. Open the P2P Connection section in the right sidebar. Your machine ID is shown at the top — a short random identifier that persists across sessions.
  2. On the machine that will initiate: click CREATE OFFER under PL LEFT (Bus F). A WebRTC SDP offer appears in the text area. Copy it and send it to the other operator by any out-of-band means — paste it in an email, a chat message, anything.
    The manual copy-paste is intentional. Loop 2.1 uses no signaling server and requires no internet infrastructure beyond the direct WebRTC connection between the two browsers. The SDP exchange is the only coordination required before the connection is fully peer-to-peer.
  3. On the second machine: paste the offer into the remote text area and click ACCEPT. An answer is generated. Copy it.
  4. Back on the first machine: paste the answer and click ACCEPT. Both machines show CONNECTED status and the Bus F strip activates.
  5. On the first machine: inject a value into Working and set Bus A source to Working, destination to Bus F outbound. The word travels to the second machine. On the second machine, set Bus A source to Bus F inbound and route it to Working.
Chain Networks
A machine can connect Bus F to one neighbor and Bus G to another, forming a chain. Machines can relay words from one side to the other, building computational pipelines across multiple physical computers. There is no hard limit on chain length — WebRTC handles the transport.

The latency display on each connection shows round-trip time in milliseconds, measured continuously. In a local network test between browser tabs on one machine this will be under 5ms; over the internet typically 20–150ms depending on geography.

Tutorial 10 Challenge Network: Chain Sum

The Challenge Network (CBX) is a coordination layer built on top of the P2P connections. It allows Challenge Boxes on different machines to run multi-computer challenges together, with the originating Challenge Box acting as the network host and verifier.

Before starting: all machines must have P2P connections established (Tutorial 09). Each machine should have the Challenge Network section open and visible.

Availability States
Each machine has three availability settings. Relay (default) — discovery probes pass through this machine transparently when it's busy. Open for Challenges — this machine will accept challenge invitations. Soft Decline — this machine is invisible to the challenge network. Set neighboring machines to Open before starting.
  1. On all participating machines: open the Challenge Network section (below Challenges in the right sidebar). Click CLOSED FOR CHALLENGES to toggle it to OPEN.
  2. On the originating machine: click ⬡ PING NETWORK. The probe travels out both connected sides, counts hops, detects ring topology, and returns. The topology display shows: 1 machine left · 1 machine right · 3 total (for a 3-machine chain).
  3. In the Challenges section, select ⬡ Chain Sum (Network) from the type dropdown. The network status line confirms machines are connected and ready.
  4. Click ▶ NEW CHALLENGE. The originating Challenge Box sends invitations to all neighbors. Each neighboring machine's Challenge Network section expands automatically showing the invitation panel.
  5. On each participating machine: click ACCEPT in the invitation panel. Once all machines accept, the origin distributes one unique value to each machine and announces which machine starts (the terminus on one side).
  6. Starter machine: your assigned value appears in the Challenge Network panel. The RELEASE button is available immediately. Click RELEASE VALUE — your value is injected into your Working Loop. This is your first operand; your incoming sum is 0, so your local sum equals your value. Route it to Bus E outbound.
    Your Challenge Box submits your local sum to the origin for verification and forwards the running sum to your neighbor automatically.
  7. All other machines: wait. Your RELEASE button is greyed out. When your neighbor's Challenge Box forwards the running sum to you, your assigned value is released simultaneously and your RELEASE button activates. Click it when ready — but not before: clicking early locks you out for 3 seconds.
  8. Add your assigned value to the incoming running sum in your ALU: Working → ALU, get both values loaded, ADD, route result back to Working. Then route the result to Bus E outbound.
  9. The chain continues until every machine has submitted. The origin verifies each submission. A wrong answer triggers an immediate full-network abort — the failing machine's position is named in the error. A clean run shows the final sum and your combined score.
The Stakes
Chain Sum fails completely on any wrong answer. Every operator in the chain bears responsibility for the result. The originating machine's progress display shows a cell per machine, going green as each one verifies. Watch it fill.
Tutorial 11 Session Logging and the .loop File

Every operator action, machine state transition, bus transfer, and challenge event is recorded in the session log. The log is readable in the sidebar and can be exported as a .loop file — a structured plaintext format with both machine-code and human-readable columns on every line.

  1. Open the Session Log section in the right sidebar. Every action since the page loaded is listed there, from most recent to oldest.
  2. Click START LOG to begin recording to an exportable session. An active session captures initial machine state, then every operator action and machine event with tick-accurate timestamps.
  3. Perform any operations — inject values, run an ALU operation, start a challenge, complete it. Each action appears in the log in real time.
  4. Click STOP LOG then DOWNLOAD .loop. The file downloads as loop21-session-TIMESTAMP.loop. Open it in any text editor.

The .loop file format has four sections: SPEC (machine parameters and constants), INIT (full machine state at session start, including all loop bit contents), OPS (every event with tick prefix), and FINAL (machine state at session end). Every line is dual-format: machine code left of ||, human-readable description right.

T000041892. BUS.A.XFER 14832 working→alu || Bus A transferred 14832 (0x39F0) from Working Loop to ALU Loop
S000000012. OP.INJECT 0100110000010100 || Operator injects 12308 (0x3014) into Injection Channel → Working Loop

T prefix lines are machine ticks; S prefix lines are operator setup actions that happen outside the tick cycle. This format is designed to be both machine-parseable for replay and human-readable for analysis.

Challenge Network Events
Multi-computer challenge events log with a CBX. prefix: CBX.CHAIN_SUM.START, CBX.CHAIN_SUM.OK, CBX.CHAIN_SUM.FAIL, and so on. A replay of a Chain Sum session shows the full discovery, invitation, assignment, and verification sequence in chronological order.

Skins

The simulator and all documentation share the same three-skin system. OG is the default light theme. Winamp is the dark green-on-black theme. Sunrise is a warm off-white theme. Skin preference is stored in localStorage and persists across sessions. The skin selector is in the top-right header on every document.
