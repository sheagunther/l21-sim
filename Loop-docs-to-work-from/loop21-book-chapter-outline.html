Chapter 1Philosophy Tract §I · Complete Record
Origin

The Minecraft castle and the throne room computer. What it felt like to build something that computed using delay lines, redstone, and a lot of patience. The question that emerged: what does computation feel like when you do all of it yourself? When there is no stored program, no automation, no abstraction between you and the data? When every addition, every comparison, every movement of a value from here to there passes through your own hands?

How that question became this machine. The move from Minecraft to a browser. Why a browser — what vibe coding made possible that would have been impossible otherwise. The AI collaboration and what it actually means.

On origins
This is the only chapter in the book that is entirely personal. It sets the register for everything that follows: this is a thing a person made because it was interesting, not because they had to.
Chapter 2Philosophy Tract (in full, restructured)
The Philosophy of the Machine

Ten ideas that shaped every design decision in Loop 2.1. Each one is brief — this is not a philosophy text, it is a design rationale. But each one will pay dividends when the reader hits the specific machine feature it explains.

  • I. Transparency — the machine hides nothing. Every bit, visible.
  • II. Flow — data moves, always. The loop is not storage. It is motion.
  • III. Control — the operator is the program. There is no other program.
  • IV. Circles — patience as a feature. The loop brings data back around.
  • V. Craft — fluency is earned, not given. Par exists for a reason.
  • VI. Slowness — you can always slow down and see exactly what is happening. At 1 Hz, nothing is hidden.
  • VII. Augmentation — this machine makes conventional computers more legible, not less necessary.
  • VIII. Computation without stored programs — what changes when you remove the instruction fetch.
  • IX. Joy as sufficient justification — it does not need to be useful. It is interesting. That is enough.
  • X. "The Operator Is the Program" — what this means, exactly, and why it is the title of the machine.

Part II
THE MACHINE

Chapter 4Manual §2 · Technical Reference §BPW
Data — The 17-Bit Word

Everything about the fundamental unit of the machine. The 17-bit word format: 1 marker bit followed by 16 data bits. Why 17 and not 16 — the marker bit as word boundary. Binary representation. Unsigned 16-bit integers, range 0–65535. Two's complement as an operator convention, not a hardware mode.

The zero-word and why it is not the same as an empty loop position. A loop position with a zero-value word has its marker bit set. An empty loop position has nothing at all. This distinction matters every time you work with zero as a value.

The most important number
BPW = 17. One marker bit + 16 data bits. This number shows up in the bus latency formula, the inject channel length, and everywhere else timing matters.
Chapter 5Manual §3 · Technical Reference §loop constants
The Four Loops

Each loop is a shift register — a sequence of bit positions that advances by one on each clock tick. Data circulates.

  • Working Loop — 18 words / 306 bits. The primary workspace. Where values are injected, where ALU results return, where scratch captures happen.
  • ALU Loop — 24 words / 408 bits. Shared with ALU operations. Values feed into ALU registers as they circulate past the capture point.
  • Memory Loop — 24 words / 408 bits. Feeds the 16-slot memory system.
  • Big Loop — 48 words / 816 bits. Home of the PM/TG filtering pipeline. The longest revolution. The most patience required.

The read head (R), write head (W), and gate (G) on each loop. Gate in HOLD mode vs DESTRUCTIVE mode. Pausing a loop vs closing its gate. STEP mode for single-tick advancement.

The key insight
Data does not sit in a loop. It moves through a loop. The read head is a window, not a destination. This is the conceptual inversion that takes new operators the longest to internalize, and it underlies everything else.
Chapter 6Manual §4 · Technical Reference §tick model
The Clock

The tick as the atomic unit of time. Clock speed in Hz — the full range, the slider, the number input. At 1 Hz you can watch a single bit travel. At 24 Hz the machine is the right speed for most work. At 60+ Hz it becomes a blur.

Recommended operating speed: 24 Hz. The reasoning: fast enough for efficient sessions, slow enough that the machine's operation is still comprehensible. Not a rule — a starting point.

The Op Count system: set a number, pick a trigger, the machine halts when the count is reached. More useful than it sounds. It lets you run the machine unattended for a precise number of operations and then examine what happened.

Chapter 7Manual §5 · Technical Reference §INJ_N
Loading Data — The Inject Channel

How values get into the machine. The inject channel: type a value using the binary switch panel, press INJECT, the word enters the Working loop at the write head. The RNG inject. The zero-pulse technique — injecting 0 as a word boundary marker between values in a stream.

The inject channel length: INJ_N = 17 — exactly one word in flight at a time, matching BPW. What happens if you inject while the Working loop is paused. Back-to-back injects and ordering guarantees.

The zero-pulse
Inject 0 between values to create a natural separator. A circulating zero-value word has its marker bit set — it is a valid word that means "boundary." The operator who understands this has unlocked stream framing.
Chapter 8Manual §6 · Technical Reference §bus table
The Buses — Moving Data

The cost of a bus transfer: 41 ticks. Always. The derivation: BUS_N (24) + BPW (17) = 41. The word enters the bus at tick 1 and exits at tick 41. Multiple words can be in the bus simultaneously. The bus is a pipeline, not a wire.

The Four Local Buses (A, B, C, D)

All four are identical in capability. This is worth stating clearly because the original documentation got it wrong — Bus C and D were incorrectly described as lesser. They are not. Bus colors, sidebar controls, inline loop header buttons. Source options, destination options (including TG threshold registers and FILE SAVE).

Bus E — The Challenge Bus

The challenge interface. Bidirectional: input values arrive, output values leave. Auto-activates when a challenge starts. The operator does not configure Bus E.

Buses F and G — P2P Connections

Left (F) and right (G) direct peer-to-peer connections over WebRTC. Real network transmission. Latency is RTT-dependent, not 41 ticks.

Buses H and I — Routed Network Buses

Route through the CBX network to named peers, including non-adjacent machines. Covered in depth in Chapter 19.

Mid-transfer bus off
Turning a bus off mid-transfer does not corrupt or duplicate data. Words in transit drain cleanly. Test this once at 1 Hz to see it happen.
Chapter 9Manual §7 · Technical Reference §ALU ops
The ALU — Computing Things
The Registers

Four registers: A, B, C, D. A and B are operands. C holds the result. D is auxiliary. ADVANCE mode (A→B→C→D after each capture) vs HOLD mode. AUTO-WRITEBACK: results automatically returned to the ALU loop.

The Operations

Every operation that exists on this machine:

  • ADD — A + B → C, Carry flag on overflow
  • SUB — A − B → C, Carry and Sign flags
  • AND / OR / XOR / NOT — bitwise, exactly as expected
  • SHL — shift left 1 (×2); top bit exits to Carry
  • SHR — shift right 1 (÷2 floor); bottom bit lost
  • NEG — two's complement: 65536 − A for nonzero; 0 for zero
  • INC — A + 1 → C; wraps 65535 → 0 with Carry
  • DEC — A − 1 → C; wraps 0 → 65535

Operations that do not exist: MUL, DIV, MOD, ABS, CMP, CLR. Knowing what's missing is as important as knowing what's there.

The Flags
  • Zero (.Z) — result = 0
  • Carry (.C) — unsigned overflow, or bit shifted out of SHL/SHR
  • Overflow (.V) — signed overflow (sign bit flips unexpectedly)
  • Sign (.S) — bit 15 of result is set
  • Parity (.P) — even number of 1-bits in result (0 bits counts as even)
Edge cases that will surprise you
NEG 0 = 0 (not 65536). NEG 32768 = 32768 (its own two's complement; Overflow fires). INC 65535 wraps to 0. DEC 0 wraps to 65535. SHL 32768 produces 0, Carry fires. Every one of these is worth testing once before you rely on them.
Chapter 10Manual §8
Memory Slots

Sixteen memory slots, addressed 0–15. Write mode, read mode, addr-read mode. Auto-increment. Batch Write All.

ADDR-READ mode: the top 4 bits of the circulating word are interpreted as a slot address. The word is consumed and the contents of the addressed slot are ejected. The data chooses the slot. When you see this clearly it is a small revelation: this is what RAM does. 4 bits. 2⁴ = 16. The derivation is not an accident.

Load is a full overwrite
Loading a file or doing a Batch Write loads all 16 slots. Whatever was there before is gone. Check your memory state before any load operation.
Chapter 11Manual §9
Working Scratch Registers

Four fast local holding slots attached directly to the Working loop. No bus transfer required — no 41-tick cost. Values captured from or sent to Working stay within the Working loop's local logic.

CAPTURE mode: arm it, choose a slot (0–3), wait for the next word at the Working read head. One-shot — fires once and disarms. The cost hierarchy: scratch (no bus latency) → memory slots (loop circulation only) → bus transfer (41 ticks). For holding an intermediate value for a few ticks, scratch is the right tool.

Chapter 12Manual §10 · Technical Reference §triggers
Counters

The counter accumulates a value as specific machine events fire. 25 trigger types total — the previous documentation was missing 11 of these. They are all here.

  • Working loop: 6 triggers (read events, write events, 4 scratch captures)
  • ALU loop: 4 triggers (read, write, ALU fire, ALU writeback)
  • Memory loop: 4 triggers (read, write, slot write, slot read)
  • Big loop: 11 triggers (read, write, PM1 match/miss/eject, PM2 match/miss/eject, TG1 pass/eject, TG2 pass/eject)

COPY vs DESTRUCTIVE eject. Counter overflow at 65535. The counter as a halt condition via Op Count.

Chapter 13Manual §11 · Technical Reference §eject indices
The Pattern Matcher

Sits on the Big Loop. Inspects every circulating word against a configured pattern. Does this automatically — always watching, every word, every revolution.

Mask register: which bits to check. Pattern register: what those bits should equal. If (word AND mask) == (pattern AND mask), the word matches. MASK = 0x0000: every word matches. MASK = 0xFFFF: only exact matches pass.

COPY eject: matched word ejected AND stays in Big Loop. DESTRUCTIVE eject: matched word ejected and removed.

Rewrite mode: matched words are modified in place before continuing. The rewritten word circulates onward — TG1 sees the rewritten value, not the original.

PM2 CASCADE: only words PM1 matched reach PM2. The key edge case: PM1 matched, PM2 rejected. The word is gone — neither in Big Loop nor at the output. Cascade is destructive at every stage.

Pipeline positions
PM1 at bit position 359 in Big Loop. PM2 at 342. TG1 at 325. TG2 at 308. Spaced 17 bits apart — exactly one word width. The pipeline is physical.
Chapter 14Manual §12
The Threshold Gate

Where the Pattern Matcher asks "does this word match a pattern?", the Threshold Gate asks "is this word above or below a number?" GTE, LTE, and EQ modes. COPY vs DESTRUCTIVE eject. CLAMP mode: values that would be ejected are replaced by the threshold value instead.

LOAD FROM BIG: the next word to pass becomes the new threshold. Useful for dynamic threshold setting from loop data.

TG2 CASCADE + TG1 GTE + TG2 LTE: a band-pass filter. Only values in the range [TG1 threshold, TG2 threshold] survive. Both stages in one configuration. No code. Just setup.

The full four-stage pipeline
PM1 → PM2 → TG1 → TG2. All four running simultaneously on Big Loop. Every circulating word passes through all four stages automatically. Configure each stage, run the loop, step back. This is the most hardware-like thing the machine does.
Chapter 15Manual §18 · FTP chapter
The File System

The machine's one concession to persistence. A named storage space that survives between sessions.

What a file is: exactly 16 × 16-bit values. The complete memory slot state, named. No header, no checksum, no metadata beyond the filename. One file = 256 bits. Full /local (8 files) = 2,048 bits of persistent state.

The null-slot invariant: empty slots are saved as zero-value words. On load, all 16 slots are always written, including zeros. Position is preserved by sequence — word N always maps to slot N. Null slots can never be skipped.

Building a file: FILE SAVE as a bus destination (any of A–D). Batch Write All streams 16 words through the bus at current clock speed. At 1 Hz you can watch every bit cross. The cost is real.

The .l21x archive: plain text, human-readable, editable. Two rows of eight hex entries per file. Round-trippable: save → export → edit in a text editor → import → load → correct values.

Chapter 16FTP chapter (in full)
FTP — Transferring Files Between Machines

The FTP system allows an operator to send a named file from their /local directly to another operator's /local. The transfer is not observable as a bit-serial bus operation. This chapter explains why that is acceptable.

Why FTP works differently
The machine has two layers: computation (physical, observable, costs ticks) and coordination (abstract, automatic — the infrastructure that makes multi-machine work possible). CBX invitation packets, hub election, heartbeats — none of these traverse a loop. FTP is in the coordination layer. The physical cost of a file was paid when it was built. FTP delivers something that was already made.

The four messages: FTP_OFFER (sender → recipient, includes the full 16-slot payload), FTP_ACCEPT, FTP_DECLINE, FTP_CANCEL. Sequence diagrams, failure modes, and routing through named networks all covered.

What FTP is not: a way to move raw data without paying the cost. You cannot FTP values circulating in a loop. You cannot FTP a partial memory state. Only named, complete, 16-slot snapshots built through the normal file pipeline.

Part III
THE NETWORK
"When one machine isn't enough."
On the network
A single Loop 2.1 machine is a meditation device for one operator. A connected network of Loop 2.1 machines is a distributed computer. The same principles apply. The same costs apply. Everything is just bigger.
Chapter 17Manual §13 · Networking Guide
P2P Connections

Direct peer-to-peer connections between two machines via WebRTC. No server involved. Bus F (left / PL) and Bus G (right / PR).

The SDP handshake: CREATE OFFER → copy → send to peer via any channel → peer pastes and accepts → answer comes back → CONNECTED. Most first-time failures are because someone clicked in the wrong order.

Latency: WebRTC RTT rather than 41 ticks. On a local network: under 5ms. Over the internet: 20–150ms. The 41-tick cost model no longer applies to F and G.

The inbound buffer: words arrive from the peer and queue if the receiving loop isn't ready. Buffer depth is observable. What happens when it fills.

Chapter 18Manual §15 · Networking Guide
Named Networks

Groups of connected machines with a shared identity and a hub. Hub-and-spoke topology. Members communicate through the hub.

Hub election: if the hub disconnects, remaining members elect a new hub after ~15 seconds. Bully algorithm — first machine to claim hub status wins. With two machines, the election is immediate.

The all-decline edge case
Hub invites all machines; all decline. Hub forms a network of one. Heartbeat fires to nobody. No crash, no error. This is correct behavior.
Chapter 19Manual §15 · CBX Protocol Spec
Buses H and I — Routed Network Buses

Unlike F and G (direct P2P), Buses H and I route to named network members — including machines not directly connected. They are the long-distance buses.

Routing is transparent: intermediate machines forward the word without receiving it. The operator acting as a relay can have H targeted at one machine and I targeted at another simultaneously — routing words from one peer to another. The operator is the router.

Chapter 20Manual §14 · CBX Protocol Spec v.203
The Challenge Network (CBX)

The Challenge Bus eXchange — the coordination protocol that makes multi-machine challenges possible. CBX handles logistics, not computation. The computation is always done locally by each operator.

The Chain Sum sequence: INVITE → ACCEPT/DECLINE → ASSIGN (each machine gets its assigned value and chain position) → GO → SUBMIT (each non-origin machine sends its computed running sum) → SUM_OK or SUM_FAIL → COMPLETE or ABORT.

If any SUBMIT is wrong: SUM_FAIL fires with the exact position and discrepancy. The origin can verify any submitted sum independently because it was assigned all values at the start.

The Network Monitor: every CBX packet visible in real time. Capture, modify, and inject packets to test protocol behavior. Set a packet's TTL to 0, inject it, watch it die. This is how you learn what the protocol does by watching it fail.

Part IV
THE LANGUAGE
"Writing down what the machine does."
Two languages
The session log is what actually happened. Loopscript is what you intended to happen. They are complementary, not redundant. The log is history. Loopscript is a plan.
Chapter 21Technical Reference §5 · Manual §16
The Session Log — .loop Format

Five sections: SPEC, INIT, OPS, FINAL, STATS.

  • SPEC — machine configuration at session start. Known issue: SPEC.BUSES writes 2 with 34-slot pipelines. The machine has 9 buses with 24-slot pipelines. The SPEC section is not currently accurate. Know this.
  • INIT — initial bit state of all four loops
  • OPS — every logged event. T###. prefix for tick events, S###. prefix for operator actions. Dual-format: machine code || human description on every line.
  • FINAL — final bit state of all four loops
  • STATS — 20 statistics keys covering every countable thing the machine tracked during the session
Chapter 22Loopscript Specification v.210v4 (in full)
Loopscript — .lps Format

The notation for describing L21 algorithms. Not executed by the machine — read by operators. A plan, a recipe, a score.

Three sections: HEADER (title, author, date, variant, IN/OUT descriptions), SETUP (non-default configuration — silence means default), EXEC (the algorithm as operator action sequence).

EXEC syntax: one action per line. Branching: .Z?→ {} :→ {}. Loops: #N {}, UNTIL(cond) {}, #CTR {}. Parallel: |. Labels: ::LABEL. WAIT() as operator-action marker.

Multi-machine Loopscript: MACHINE: N blocks. WAIT(CBX.GO), WAIT(EIN) — these are operator-action signals. The operator waits for the visual cue and proceeds. The machine does not wait autonomously.

Four worked examples: Add Two Numbers (minimum viable), Filter (SETUP doing real work), Find Maximum without ALU (TG destructive mode), Selection Sort (memory + nested loops + counter + branching).

Chapter 36Course Outline v.264v2 (in full)
Loop 2.1 in the Classroom

The 15-week course structure. Eight modules, each with a theme, a two-week arc, CS concepts introduced, and challenge work. The guiding principle: every CS concept is encountered on the machine before it is formalized in lecture.

Binary representation, overflow, masking, state machines, sorting algorithms, complexity, pipelining, network protocols, distributed algorithms — all of them arrived at by experience first, formalized second.

Grading structure: challenge scores (35%), written reflections (20%), midterm timed session (20%), final project (25%). The final project synthesis question: "Name one thing the machine taught you that a textbook could not have."

CS topic cross-reference table: every topic, when students first encounter it experientially, when it is formalized. 30 topics tracked from Week 1 through Week 14.

Chapter 38Design document (this session)
The Instructor Dashboard (Proposed)

A monitoring and assessment tool that does not yet exist as a shipping product. Included here because the design is complete enough to serve as a specification, and because the simulator hooks needed to support it are worth documenting now.

The Live Monitor: fleet view (real-time miniaturized machine state for every student), zoom view (read-only mirror of one student's machine), rewind (scrubbing backward through event stream), class-wide aggregate (par distribution, common stuck points by structural similarity).

The Grader: session log compliance (SPEC accuracy, tick continuity, timestamp plausibility), integrity analysis (action sequence fingerprinting, temporal analysis, reference solution comparison), structured report per submission.

Hooks needed in the simulator: session metadata header (machine ID, operator ID, class ID), continuous WebRTC event streaming, machine state snapshots, challenge boundary events, instructor API, classroom mode indicator.
